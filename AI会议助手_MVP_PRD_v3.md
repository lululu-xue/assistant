## 一、产品概述

### 1.1 产品名称

AI Meeting Assistant（AI会议助手）

### 1.2 产品定位

一款面向个人办公场景的 AI 会议工作助手，围绕**会前准备、会后整理、Todo 沉淀**三个核心场景，帮助用户降低会议沟通成本与工作同步成本。

### 1.3 产品核心价值

**会后**：用户上传会议纪要后，AI 自动总结会议重点、提取待办（含责任人与时间节点）、识别风险与卡点，并把待办自动沉淀进个人 Todo 列表。

**会前**：AI 读取某个 tag 下最近的会议、未完成 Todo 与当前风险，自动生成一份「本次建议汇报内容」，用户可微调后一键复制 / 导出。

### 1.4 产品边界（MVP 不做）

本产品定位为**个人工作助手**，不是企业协同平台。MVP 阶段明确不做：多用户协同、企业权限体系（复杂 RBAC）、实时聊天、企业微信 / 飞书 / Slack 集成、实时会议转写、音视频转写、WebSocket 协同。

> 边界目的：保证 Claude Code 能在可控范围内快速完成，并让 demo 聚焦在「AI 会议工作流」这一核心亮点上。

## 二、目标用户

核心用户为高频参会的个人贡献者与负责人，MVP 以**项目经理**为第一优先 persona。

**项目经理（第一优先）**：典型场景为周会、项目同步会、风险评审会、跨部门沟通。痛点：会前不知道怎么汇报、会后整理耗时、Todo 容易遗漏。

**产品经理**：需求评审、项目同步、开发跟进。

**开发 / 设计**：Standup、Sprint 同步、风险同步。

> demo 数据建议围绕「项目经理 + 单个项目」构造，使会前汇报上下文自然连贯。

## 三、核心用户流程

### 3.1 会后流程

用户进入「上传会议」页 → 选择 tag（或留默认 `未分类`）→ 上传 / 粘贴会议纪要 → AI 自动解析 → 输出结构化结果（总结 / 进展 / Todo / 风险 / 时间节点）→ 用户可在结果页编辑、删除、手动补充 → Todo 自动写入数据库并进入 Dashboard。

### 3.2 会前流程

用户进入「会前准备」页 → 在顶部选择要汇报的 tag（含「全部」）→ 点击生成 → AI 基于该 tag 下的上下文生成「AI 汇报卡」→ 用户编辑微调 → 一键复制 / 导出 Markdown / 导出文本。

### 3.3 tag 如何贯穿全流程

tag 在「上传会议」时确定，并在 AI 提取时**复制**到该次会议产生的所有 Todo 与风险上（`meeting_id` 关联 + `tag` 冗余字段）。Dashboard 与会前汇报均可按 tag 筛选。

> 简化约定：tag 在创建时复制到子记录；事后修改会议 tag **不**回溯更新已生成的 Todo / 风险（MVP 不做级联，避免复杂度）。

## 四、MVP 功能范围

每个功能给出「功能说明 → MVP 要求 → 验收标准」。验收标准是 Claude Code 完成后可逐条勾验的清单。

### 4.1 用户登录

**功能说明**：邮箱 + 密码登录，基于 Supabase Auth。

**MVP 要求**：邮箱注册、邮箱 + 密码登录、退出登录、登录状态保持（刷新不掉线）。

**验收标准**
- 新邮箱可注册并立即登录（无需邮件确认链接）。
- 已注册邮箱用错误密码登录会被拒绝并提示。
- 刷新页面后仍保持登录态。
- 退出后访问受保护页面会被重定向到登录页。

### 4.2 上传会议纪要

**功能说明**：用户上传或粘贴会议内容，并为本次会议选择一个 tag。

**MVP 要求**
- 支持文件：`txt`、`markdown`、`docx`。
- 支持交互：拖拽上传、点击上传、直接粘贴文本。
- **tag 选择**：受控下拉（选已有 / 新建），单选，默认 `未分类`。
- 可选填「会议日期」`meeting_date`，默认取上传当日（作为相对时间换算基准）。
- 上传 / 粘贴成功后自动调用 AI 分析。

**MVP 不支持**：pdf、音频、视频、实时录音。

**验收标准**
- 三种文件类型与粘贴文本均能成功读入并进入分析。
- 未选 tag 时会议落库 tag 为 `未分类`。
- 选了不存在的新 tag 名时，该 tag 进入「已有 tag」列表，下次可直接选。
- 上传后自动跳转「AI 处理中」状态。

### 4.3 AI 会议总结

**功能目标**：从会议文本中自动产出结构化摘要。

**AI 输入**：会议正文 + `meeting_date`（用于时间换算）。

**AI 输出结构**（结构化存储，详见第十、十一节）
- `summary`：一段话会议摘要
- `progress`：当前进展（列表）
- `conclusions`：核心结论（列表）
- `time_nodes`：时间节点（每项含原文与标准化日期）
- `todos`：见 4.4
- `risks`：见 4.5

**示例**

输入：`支付功能已联调完成，下周开始测试。`

输出（摘要 / 进展部分）：
- 摘要：支付功能联调完成，即将进入测试阶段。
- 进展：支付功能已完成联调；下周进入测试。

**验收标准**
- 任意会议文本都能返回上述 6 块结构；无对应内容的块返回空数组而非报错。
- 摘要、进展、结论在结果页以卡片展示，且**可编辑、可删除、可手动新增条目**。

### 4.4 AI Todo 提取（核心功能）

**功能目标**：自动识别待办事项、责任人、截止时间。

**AI 输出结构**（每条 Todo）
```json
{
  "task": "完成支付接口测试",
  "owner": "张三",
  "deadline_text": "下周五",
  "deadline_date": "2026-05-29",
  "risk_level": "medium"
}
```
- `owner`：责任人；无法识别时为 `null`。
- `deadline_text`：原文表达，始终保留。
- `deadline_date`：以 `meeting_date` 为基准换算的 ISO 日期（`YYYY-MM-DD`）；无法换算时为 `null`。
- `risk_level`：`low` / `medium` / `high`，由 AI 初判，用户可改。

**MVP 要求**
- 支持单次提取多条 Todo。
- 提取结果自动写入 `todos` 表，并继承会议的 `tag` 与 `meeting_id`。
- Todo 状态管理：`open` / `done`（可在 Dashboard 切换）。
- **可编辑**：修改任务文本、责任人、`deadline_text`、`risk_level`、状态；可删除错误条目；可手动新增一条 Todo。

**验收标准**
- 含多条待办的会议能拆出多条 Todo，各自带 owner 与两种 deadline 字段。
- 相对时间（如「下周五」）能换算成正确的 `deadline_date`（以会议日期为基准）。
- 编辑 / 删除 / 新增后，数据持久化且 Dashboard 同步。

### 4.5 AI 风险识别

**功能目标**：识别时间风险、依赖风险、阻塞问题。

**AI 输出示例**
- 测试资源尚未确认（依赖风险）
- API 接口依赖存在延期风险（时间风险）

**MVP 要求**
- 风险写入 `risks` 表，继承 `meeting_id` 与 `tag`，含 `level`（`low`/`medium`/`high`）。
- **可编辑**：改内容、改等级、删除、手动新增。

**验收标准**
- 会议中的风险点能被识别并落库。
- Dashboard 风险卡片能按等级展示，高风险用风险色高亮。

### 4.6 时间节点提取与标准化

**功能目标**：识别「下周 / 本月底 / 周五前 / 明天」等时间表达，并标准化为真实日期。

**MVP 实现**
- 第一阶段（MVP）：在 Prompt 中提供 `meeting_date` 作为基准，**让 AI 直接输出 `deadline_date` / `time_nodes[].date`（ISO 格式）**。
- 兜底（可选）：对 AI 给出的相对表达，前端可用 `chrono-node` + `dayjs` 二次校验 / 兜底解析。
- `time_nodes` 存入 `meetings.structured`（jsonb），仅作展示；其中可执行的待办时间体现在 Todo 的 `deadline_date` 上。

**验收标准**
- 「下周五」「本月底」等能换算为正确 ISO 日期；无法换算的保留原文、日期置 `null`。

### 4.7 Todo Dashboard（核心页面）

**页面目标**：集中展示我的 Todo、风险事项、最近会议，并支持按 tag 筛选。

**页面模块**

*顶部*：欢迎语 + 今日 / 待办 Todo 数量 + **tag 筛选器**（含「全部」）。

*Todo 列表*（卡片式）：字段含 任务名称、责任人、状态、`deadline_text`、`deadline_date`、风险等级。支持按状态、按 deadline 排序。

*风险卡片*：展示高风险事项与「即将延期 / 已逾期」事项。
- 「即将延期 / 逾期」用 `deadline_date` **真实计算**（如 `deadline_date <= 今天+3 天` 视为即将延期，`< 今天` 且未完成视为逾期）；无 `deadline_date` 的归入「无明确时间」分组，不参与日期排序。

*最近会议*：最近上传的会议及其 AI 总结。

**验收标准**
- 切换 tag，Todo / 风险 / 最近会议三块联动筛选。
- 「即将延期 / 逾期」分类与真实日期一致。
- Todo 状态、内容编辑后即时反映。

### 4.8 AI 会前汇报（核心亮点）

**功能目标**：帮用户「30 秒生成本次会议汇报内容」。

**交互**：会前页顶部选择 tag（含「全部」）→ 点击生成。

**AI 上下文**（按所选 tag 过滤）
- 该 tag 下**未完成**的 Todo
- 该 tag 下**未关闭**的风险
- 该 tag 下**最近 3 次**会议的总结与进展

**AI 输出（AI 汇报卡）**
- 本周进展
- 当前风险
- 当前卡点
- 本次建议重点同步内容
- 下周计划

**存储与编辑**：生成后存为**可编辑草稿**到 `reports` 表（含所选 `tag`、`title`、`content`、`updated_at`）。用户可在页面内编辑后再导出，历史草稿可重新打开。

**页面操作**：一键复制、导出 Markdown、导出纯文本。

**示例（节选）**
- 本周进展：已完成登录页开发；已完成需求评审。
- 当前风险：测试资源尚未确认。
- 本次建议同步：支付功能已进入联调阶段；当前测试资源存在风险。

**验收标准**
- 选定 tag 后能生成五段式汇报卡，内容来自该 tag 下真实数据。
- 生成结果可编辑、可保存为草稿、可再次打开。
- 三种导出方式均可用。

## 五、AI 输出可编辑性（横切要求）

所有 AI 产出默认是「草稿」，用户对其拥有最终控制权。**会议总结、进展、结论、Todo、风险、会前汇报**均需支持：编辑文本、修改关键字段（责任人 / deadline / 等级 / 状态）、删除错误条目、手动新增条目。

> 这是产品的核心理念，也是面试讲述点：**AI 给草稿，人做决策**——而非让 AI 不可纠错地替用户拍板。

## 六、错误处理与边界情况（横切要求）

**AI 调用失败 / 返回非合法 JSON**
- 后端对 AI 输出做 JSON 结构校验；校验失败**自动重试 1 次**。
- 仍失败时，前端展示友好错误（如「AI 分析失败，请重试」）+ 重试按钮，不白屏、不丢用户已上传内容。

**空状态**
- 无任何会议 / Todo 时，Dashboard 与会前页展示引导空状态（如「先上传一份会议纪要试试」+ 引导按钮）。

**输入异常（轻量处理）**
- 空内容 / 过短内容：提示「内容过短，无法分析」。
- 超大文本：做长度上限提示（具体阈值可由实现决定，MVP 给出友好截断或拒绝即可）。

## 七、页面设计

### 7.1 登录页
风格参考飞书登录页。元素：Logo、产品名、邮箱 + 密码输入、登录 / 注册切换、提交按钮。UI：居中布局、大留白、浅灰背景、蓝色主按钮。

左右结构。左侧导航：Dashboard、上传会议、会前汇报、Todo、设置。右侧主区域：顶部欢迎语 + 待办数 + tag 筛选器；Todo 区（卡片）；风险区（红色风险 Tag、即将延期 / 逾期分组）；最近会议（AI 总结）。

卡片化展示：会议总结、进展 / 结论、Todo 列表、风险列表、时间节点。每块均可编辑 / 删除 / 新增。

## 十一、Prompt 设计

> 共同约束：要求模型**只输出合法 JSON**（无多余解释）；后端解析失败自动重试 1 次。

### 会议总结 + 抽取 Prompt（会后）
```txt
你是 AI 会议助手。今天的会议日期是：{meeting_date}（用于把相对时间换算成具体日期）。

请阅读以下会议内容，输出严格的 JSON（不要任何额外文字），结构如下：
{
  "summary": "一段话会议摘要",
  "progress": ["当前进展1", "当前进展2"],
  "conclusions": ["核心结论1"],
  "time_nodes": [{"text": "原文时间表达", "date": "YYYY-MM-DD 或 null"}],
  "todos": [
    {"task": "", "owner": "责任人或null", "deadline_text": "原文或null",
     "deadline_date": "YYYY-MM-DD 或 null", "risk_level": "low|medium|high"}
  ],
  "risks": [{"content": "", "level": "low|medium|high"}]
}

规则：
- 所有相对时间（如「下周五」「本月底」）以 {meeting_date} 为基准换算成 deadline_date / date；无法确定时填 null。
- 无对应内容的字段返回空数组。
- 严格输出 JSON，不要 Markdown 代码块包裹。

会议内容：
{meeting_content}
```

### 会前汇报 Prompt（会前）
```txt
你是 AI 会议汇报助手。请基于以下「{tag}」相关的上下文，生成一份适合口头/书面同步的简洁汇报。

未完成 Todo：{open_todos}
当前风险：{open_risks}
最近 3 次会议总结：{recent_summaries}

请输出严格 JSON（不要额外文字）：
{
  "weekly_progress": ["..."],
  "current_risks": ["..."],
  "blockers": ["..."],
  "suggested_sync": ["本次建议重点同步内容"],
  "next_plan": ["下周计划"]
}
内容要简洁、可直接用于汇报，避免空话。
```

## 十二、Claude Code 开发顺序

1. **Phase 1**：Next.js 初始化 → Supabase 连接 → 建表（含本版 schema）→ 邮箱 + 密码登录页与鉴权。
2. **Phase 2**：上传会议页（含 tag 受控下拉、会议日期）→ 会议总结 Prompt 接入 → AI 结果页（卡片 + 可编辑）。
3. **Phase 3**：Todo 自动写入（含 owner、双 deadline、tag 继承）→ Dashboard（tag 筛选、即将延期 / 逾期真实计算）。
4. **Phase 4**：AI 会前汇报（tag 上下文 → 生成 → 可编辑草稿 → 导出）→ 风险识别完善 → 错误兜底与空状态收尾。

---

## 十三、MVP 成功标准

**功能完成**：用户能上传会议、获得结构化 AI 总结、自动生成可编辑 Todo、按 tag 生成并导出会前汇报。

**技术完成**：Claude Code 能独立生成前端页面、完成 Supabase CRUD（含本版 schema 与关系）、完成 Claude API 调用与 JSON 校验兜底。

**产品完成**：用户能把它真正用于一次日常会议的「会后整理 + 会前准备」闭环，且 AI 出错时可自行纠正。

## 十四、后续迭代方向

**Phase 2（产品向）**：音频 / 视频转文字、AI 周报 / 日报。

**Phase 3（集成向）**：飞书 / Slack 接入、日历同步、自动提醒；如需多项目，可由 tag 平滑升级为 `projects` 表。

---

## 十五、总结

AI Meeting Assistant 的核心目标不是「替代会议」，而是「帮用户更轻松地完成会议工作流」。产品重点体现：AI 上下文理解、工作连续性、Workflow 能力、Todo 自动沉淀、会前智能准备，以及**「AI 给草稿、人做决策」的可控性**。

适合作为 **AI Coding + Workflow + Personal Agent** 类型的面试 / 作品集项目。
